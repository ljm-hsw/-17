## 1. 设计目标

本任务不是编写普通项目介绍，而是制作能够直接用于竞赛提交的正式作品设计文档。文档必须同时完成三类论证：

1. **产品全闭环**：说明游客如何从微信登录、绑定 RFID/NFC 手环、现场打卡，连续完成地图导览、路线回顾、智能推荐与个性化游记生成；说明运营人员如何通过 Web 管理端维护点位、设备、卡片及追溯数据。
2. **AIoT 技术实现**：说明 ESP32-S3 感知终端、RFID/NFC 数据采集、设备认证、事件幂等、Django 数据服务、四端接口协作、Dify/云端大模型受控调用与安全设计。
3. **文旅叙事价值**：说明系统如何把真实空间中的游览行为转化为可理解、可回顾、可分享的个人文化记忆，并论证其在校园、景区、古镇与乡村文旅场景中的应用价值。

三条线必须在同一条“真实感知—可信数据—智能服务—文化记忆”主线中相互支撑，不拆成互不相关的产品、技术和宣传材料。

## 3. 内容定位与写作口径

### 3.1 系统状态口径

作品设计文档描述最终提交版本的完整设计与实现效果，不写成开发周报，也不在正文区分“当前已完成”和“正在联调”。提交前的工程开发、测试和演示必须反向对齐文档中承诺的最终效果。

正文中的功能只包含最终参赛系统范围：

- ESP32-S3 + RFID/NFC 实体点位打卡；
- 微信登录、手机 NFC/手输 UID + 一次性激活码绑卡；
- 一个账号可绑定多张卡，一张卡同一时间只属于一个账号；
- 以中国标准时间自然日组织游览会话；
- 小程序地图、点位、进度、记录、照片、推荐、AI 导游与游记；
- Django 后端、PostgreSQL 数据库与受控 REST API；
- Vue 3 Web 展示与运营管理端；
- Dify/云端大模型受控工具与事实校验工作流；
- 摄像头照片仅在明确标识、知情授权的拍照点与有效打卡关联。

拍照点采用“游览前知情授权 + 有效打卡 + 小程序即时提示 + 用户确认触发”的四重约束。游客在开始游览前通过小程序了解拍摄目的、使用范围和访问方式并作出授权；到达摄像头拍照点后，系统先完成 RFID/NFC 打卡验证，再由小程序提示即将拍照，游客确认后才触发摄像头。后端签发与该次 `event_id` 绑定的短时拍照授权，照片上传后保存授权版本和事件关联。系统不连续采集影像，也不允许摄像头脱离有效打卡独立形成游客轨迹。

UHF 无感感知、多 Agent、社交匹配等不作为系统现有功能展开。应用前景只讨论已设计架构向其他文旅场景复制的价值，不通过堆叠未实现功能制造创新感。

### 3.2 核心论点

文档应持续证明以下结论：

1. “游迹织梦”不是普通电子地图，而是由真实物联网感知驱动的个性化文旅系统。
2. RFID/NFC 手环不是业务数据库，也不是手机替代品，而是连接游客、点位终端与数字服务的低成本移动身份媒介。
3. 后端产生并保护可信事实；小程序和管理端消费受权限控制的数据；AI 只在受控工具和事实快照范围内生成内容。
4. 个性化游记不是脱离现场的文本生成，而是对真实游览顺序、点位知识和授权素材的文化叙事重组。
5. 以江安校区为首个场景验证闭环，但 `Scene` 数据模型、点位配置和设备归属机制支持后续低成本迁移。
6. 摄像头拍照是游客主动参与的文旅留念服务，不是身份识别或无感监控；每张照片都能追溯到用户授权和一条有效打卡事件。
