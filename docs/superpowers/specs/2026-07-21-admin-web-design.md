# 游迹织梦：Web 展示与运营管理端产品设计

> 文档状态：草案 v0.1（供团队评审）
> 适用范围：四川大学江安校区最终参赛系统
> 依赖文档：[产品与业务规则设计](2026-07-13-jiang-an-mvp-product-business-design.md)、[技术架构与数据设计](2026-07-13-jiang-an-mvp-technical-architecture.md)
> 产品定位：面向比赛展示和项目运营人员的独立 Web 管理端，不面向普通游客

## 1. 目标与边界

Web 管理端使用同一套 Django + DRF 后端，同时满足两个目标：

1. 在比赛展示中，以清晰、美观、可信的方式展示物联网打卡闭环、点位热度、设备状态和 AI 成果；
2. 在日常运营中，管理江安校区的点位、设备、卡片、绑定、游览、打卡、照片和 AI 内容。

Web 管理端不是第二个后端，不保存数据库凭据或业务密钥，不直接读取 PostgreSQL。所有数据和操作都必须经过 `/api/v1/management/*`，由 Django 完成身份验证、权限判断、业务校验和审计。

Django Admin 继续存在，但仅供系统管理员处理账号权限、异常数据和故障恢复，不作为比赛展示或日常主要运营页面。

## 2. 用户角色与权限

| 角色 | 主要职责 | 可操作范围 | 明确禁止 |
|---|---|---|---|
| 展示人员 | 比赛讲解与实时演示 | 查看展示看板、公开点位、脱敏设备和打卡统计 | 修改数据、查看敏感标识 |
| 内容运营 | 维护点位和文旅内容 | Scene、Spot、地图坐标、标签、图片、公开知识 | 设备密钥、卡片激活码、用户隐私 |
| 设备运营 | 部署和维护硬件 | Device、Spot 归属、启停、状态、凭据轮换 | 游客对话、照片正文、卡片绑定纠错 |
| 数据管理员 | 处理卡片和异常业务数据 | Card、激活码、绑定、游览和受审计的打卡纠错 | 系统级账号和环境密钥 |
| 系统管理员 | 管理系统与权限 | 所有管理功能、后台账号、角色和审计 | 绕过审计直接修改生产数据库 |

前端路由、按钮和字段应按权限隐藏或禁用，但真正的授权必须由后端再次判断。展示人员可使用只读的“演示模式”，避免比赛讲解时误操作数据。

## 3. 信息架构

```text
登录
└── 管理端主框架
    ├── 展示看板
    ├── 场景地图
    ├── 点位与内容
    ├── 设备管理
    ├── 卡片与激活码
    ├── 用户与绑定
    ├── 游览与打卡
    ├── 照片管理
    ├── AI 内容
    ├── 管理员与角色
    └── 操作审计
```

主导航按业务名称组织，不暴露 Django App、数据库表等技术概念。当前只有江安校区一个 Scene 时，页面可默认进入该场景，但 URL、筛选器和 API 仍保留 Scene 维度。

## 4. 展示看板

### 4.1 展示目标

看板应让评委在较短时间内看懂：游客如何通过手环与真实空间互动、系统如何接收可信事件、运营方如何观察现场状态，以及 AI 如何建立在真实路线之上。

### 4.2 首屏指标

| 指标 | 定义 | 数据来源 |
|---|---|---|
| 今日游客 | 当日存在 VisitSession 的不同用户数 | `VisitSession` |
| 今日有效打卡 | 当日 `accepted` 的 CheckinEvent 数 | `CheckinEvent` |
| 已启用点位 | 当前 Scene 可打卡的 Spot 数 | `Spot` |
| 在线设备 | 在约定在线窗口内上报心跳/事件的启用设备数 | `Device.last_seen_at` |
| 今日照片 | 当日可用的 CheckinMedia 数 | `CheckinMedia` |
| 今日 AI 生成 | 当日成功生成的游记/分享内容数 | `AIGeneration` |

“在线设备”的时间窗口属于硬件联调参数，正式演示前固定并展示说明。统计卡片必须标明时间范围，不能混用累计值和当日值。

### 4.3 看板模块

- 江安校区卡通地图：显示点位、设备在线/离线状态和当日打卡热度；
- 实时事件流：展示最新有效打卡的时间、点位、方式和脱敏卡片信息；
- 点位热度排行：按当前选择时间范围统计不同有效游客或有效打卡；
- 游览进度分布：展示游客完成度区间，不公开个人身份；
- 设备健康状态：在线、离线、停用和最近上报时间；
- 照片墙：仅展示已授权、状态正常的打卡照片；
- AI 成果展示：展示经过事实检查并允许展示的游记摘要或分享内容。

比赛展示默认使用真实测试环境数据。需要预置演示数据时，数据必须明确标记为“演示数据”，不得与现场真实数据混合后声称为实时结果。

### 4.4 刷新策略

第一版采用固定间隔轮询或手动刷新，不强制引入 WebSocket。默认建议每 10 秒刷新摘要与设备状态，事件流可每 5 秒刷新；最终间隔按现场网络和后端压力实测确定。页面必须显示“最后更新时间”和刷新失败提示。

## 5. 运营管理功能

### 5.1 场景地图

- 查看 Scene 基础资料和地图底图；
- 在地图上查看所有 Spot 的比例坐标、类型、状态与设备状态；
- 点击点位进入详情；
- 仅有内容运营或系统管理员可进入坐标编辑模式；
- 坐标保存前提供预览，避免误操作使点位移出地图范围。

第一版只完整运营江安校区，但不得删除 Scene 维度或把江安校区名称写死在通用组件中。

### 5.2 点位与内容

- 创建、编辑、启用和停用点位；
- 编辑名称、简介、类型、标签、建议停留时间、地图比例坐标；
- 设置是否可打卡、是否拍照点；
- 上传、排序、替换和停用点位图片；
- 编辑供小程序和 AI 使用的已审核知识内容；
- 发布前检查必要字段和坐标范围。

已产生历史打卡的 Spot 不允许物理删除，只能停用；修改名称或知识内容不得改变历史打卡事实。

### 5.3 设备管理

- 登记 `device_id`、所属 Scene、Spot、设备类型和状态；
- 查看在线状态、最后上报时间、最近错误码和固件版本；
- 启用、停用或更换设备归属；
- 首次配网或轮换时生成设备密钥；
- 明文密钥只显示一次，之后只展示指纹或末四位；
- 查看该设备最近的打卡事件和失败原因。

设备从一个 Spot 调整到另一个 Spot 时，新事件使用新归属；历史事件继续保留原 Spot，不随设备配置变化而迁移。

### 5.4 卡片与激活码

- 单张或批量登记实体卡；
- 显示脱敏 UID、状态、发放时间和当前绑定状态；
- 为实体卡生成一次性激活码；
- 支持导出适合制作/发放实体卡的卡号与激活码清单；
- 激活码明文只在生成时显示或导出一次，数据库仅保存安全散列；
- 支持撤销未使用激活码并重新生成；
- 禁止恢复或再次使用已消费的激活码。

批量导入必须先预览成功、重复、格式错误和已存在数量，用户确认后才执行写入。

### 5.5 用户与绑定

- 按脱敏信息查询用户、当前卡片和历史绑定；
- 查看绑定时间、解绑时间和操作来源；
- 处理用户遗失卡片、换卡和误绑；
- 强制解绑必须填写原因、二次确认并记录管理员；
- 解绑不删除历史游览、打卡、照片或 AI 内容。

普通运营页面不展示微信稳定身份标识全文、完整卡 UID 或令牌信息。

### 5.6 游览与打卡

- 按日期、Scene、Spot、Device、状态和打卡方式筛选；
- 查看某次自然日游览的路线、顺序、完成度和关联照片；
- 查看 CheckinEvent 的接收时间、设备时间、处理状态和失败码；
- 重复事件应明确显示为幂等重传或同点再次刷卡，不能当作额外完成度；
- 有权限的数据管理员可发起纠错，但不得直接覆盖原事件。

打卡纠错采用“追加纠错记录”的方式：保留原始事件，记录纠错类型、原因、管理员和时间，再由业务查询决定该事件是否参与进度与 AI。删除按钮不进入普通管理流程。

### 5.7 照片管理

- 按日期、Spot、状态和所属游览筛选；
- 查看缩略图、关联事件、拍照时间和授权版本；
- 对违规、错误或用户要求删除的照片执行停用/删除流程；
- 所有查看原图、停用和删除操作均需权限并记录审计；
- 非拍照点或无有效事件关联的媒体标记为异常，不能进入照片墙。

### 5.8 AI 内容

- 查看生成类型、所属用户（脱敏）、所属游览、生成状态和时间；
- 查看事实快照中引用的真实点位；
- 查看生成结果和事实检查状态；
- 对失败任务进行受控重试；
- 对不适合公开展示的内容设置“禁止展示”；
- 不允许后台人员修改用户真实路线以配合生成结果。

## 6. 高风险操作与审计

以下操作必须二次确认、填写原因并写入审计日志：

- 强制解绑或换卡；
- 撤销、重置或重新生成激活码；
- 停用设备、轮换设备密钥或变更设备 Spot；
- 对打卡事件追加纠错；
- 停用或删除打卡照片；
- 修改管理员角色与权限；
- 将 AI 内容设为可公开展示或取消展示。

审计记录至少包括管理员、角色、时间、动作、目标类型与 ID、变更前后摘要、原因和 `request_id`。审计记录不得由普通管理员在 Web 管理端删除。

## 7. 页面状态与错误处理

所有列表和看板必须覆盖以下状态：加载中、无数据、筛选无结果、无权限、请求失败和数据部分不可用。不得将请求失败显示为“0 条数据”。

具体要求：

- 表单保存失败时保留用户输入，并定位到出错字段；
- 网络超时不自动重复提交创建、解绑、密钥轮换等写操作；
- 删除、停用和解绑必须区分，按钮文案不能统一写成“删除”；
- 令牌过期时尝试安全刷新；刷新失败返回登录页，不无限循环；
- 看板某一模块失败时，其余模块继续可用，并显示该模块的独立错误状态；
- 批量操作返回逐项结果，不能只显示笼统的成功或失败。

## 8. 前端技术设计

管理端采用：

- Vue 3 + TypeScript + Vite；
- Vue Router 负责路由与权限守卫；
- Pinia 保存登录态、角色权限和必要的页面状态；
- 统一 API Client 处理后端地址、JWT、刷新、`request_id` 和错误码；
- Element Plus 承担表格、表单、弹窗和基础布局；
- ECharts 负责统计图表；
- 江安校区地图使用同一张卡通底图和同一组比例坐标数据。

UI 应以清晰稳定为主，展示看板可有更强的品牌视觉，但运营页面不得为了视觉效果牺牲可读性和操作确认。

建议路由结构：

```text
/login
/dashboard
/scene-map
/spots
/devices
/cards
/users
/visits
/checkins
/media
/ai-generations
/administrators
/audit-logs
```

## 9. 管理 API 边界

管理端只调用 `/api/v1/management/*`。小程序接口与管理接口共享业务服务和数据库规则，但使用不同权限类、序列化字段和限流策略。

| API 领域 | 示例能力 |
|---|---|
| Dashboard | 指定 Scene 和时间范围的摘要、热度、设备状态、事件流 |
| Scenes/Spots | 场景读取、点位 CRUD、发布/停用、坐标与媒体管理 |
| Devices | 设备登记、归属、启停、密钥首次发放/轮换、状态读取 |
| Cards | 卡片导入、状态、激活码生成/撤销、脱敏查询 |
| Users/Bindings | 用户脱敏查询、绑定历史、受审计的强制解绑 |
| Visits/Checkins | 游览与事件查询、追加纠错记录 |
| Media | 媒体查询、授权状态、受审计的停用/删除 |
| AI Generations | 生成记录、事实快照、重试和展示状态 |
| Administrators/Audit | 管理员、角色权限和只读审计日志 |

管理端不得复用返回游客个人数据的接口来规避管理权限设计，也不得通过前端组合多个无权限接口推断敏感数据。

## 10. 视觉与比赛展示要求

- 延续“游迹织梦”和江安校区的视觉识别，首页应能快速展示系统名称、当前 Scene 和数据时间范围；
- 展示看板优先适配常见 16:9 演示屏，同时保证笔记本浏览器可用；
- 地图、指标卡、事件流和设备状态形成清晰的信息层级；
- 在线、离线、停用、成功、失败等状态除了颜色，还需配合文字或图标；
- 展示模式隐藏编辑入口和敏感信息，可通过全屏进入；
- 页面中的“实时”“今日”“累计”等词必须与数据口径严格一致；
- 所有动画克制且不影响数值读取，不使用无法解释的装饰性数据。

## 11. MVP 验收标准

管理端达到可交付状态，至少应通过以下验收：

1. 管理员可以登录，前端路由和后端 API 均按角色限制权限；
2. 展示看板能从真实后端显示今日游客、有效打卡、点位、设备、照片和 AI 生成摘要；
3. 地图可展示江安校区点位、热度和设备状态；
4. 内容运营可以创建/编辑/停用点位并维护地图坐标和图片；
5. 设备运营可以登记设备、绑定 Spot、查看状态并安全轮换凭据；
6. 数据管理员可以登记卡片、生成一次性激活码、查询绑定并完成受审计解绑；
7. 管理员可以查询自然日游览、打卡事件、照片和 AI 生成结果；
8. 完整 UID、激活码明文和设备密钥不会在普通列表或日志中泄露；
9. 高风险操作具有二次确认、原因和审计记录；
10. 展示模式不提供写操作，断网或部分接口失败时不会显示伪造或错误的零值；
11. Django Admin 仍可供系统管理员进行内部维护，但比赛主流程无需进入 Django Admin。

## 12. 明确不在第一版范围内

- 面向多个客户的 SaaS 租户、计费或商业权限体系；
- 拖拽式大屏搭建器和任意报表设计器；
- 对设备进行远程固件 OTA 升级；
- 从管理端直接控制摄像头拍摄或进行实时视频监控；
- 大规模实时流处理、复杂客流预测或人脸分析；
- 管理员在后台代替游客生成或修改真实游览路线。

## 13. 与四端系统的关系

| 端 | 面向对象 | 核心职责 | 与后端关系 |
|---|---|---|---|
| Django 后端 | 所有客户端的可信业务层 | 数据、规则、权限、API、AI/媒体代理 | 唯一数据库访问者 |
| Web 管理端 | 比赛展示人员和运营管理员 | 展示真实数据、执行权限内运营操作 | 仅调用管理 API |
| 微信小程序 | 游客 | 登录绑卡、导览、记录、推荐和 AI | 仅调用游客 API |
| ESP32 Arduino 设备端 | 现场点位设备 | 读卡、事件上传、反馈、必要的照片上传协同 | 仅调用设备 API |

四个端共享同一套业务定义和 API 契约，不共享数据库连接、密钥或前端状态。任何端都不能绕过 Django 后端修改业务事实。
