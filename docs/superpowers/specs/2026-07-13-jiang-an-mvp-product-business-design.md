# 游迹织梦：江安校区最终参赛版产品与业务规则设计

> 文档状态：草案 v0.2（已加入独立 Web 管理端，供团队评审）
> 适用范围：四川大学江安校区最终参赛系统（初赛交付）
> 依据：项目概述（2026-07-13）
> 本文定位：将已确定的项目定位与功能范围转化为可开发、可验收的产品和业务规则；不替代后续的技术架构、数据模型、API 与硬件通信文档。

## 1. 文档使用约定

- 本文中的“必须”是本次最终参赛版的交付要求；“可选”不影响本次演示闭环。
- 本文没有改变项目概述所确定的产品方向。后续若出现项目概述未决定、但实现必须依赖的事项，须先记录为“待确认项”，不得由任一开发成员自行假定为最终规则。
- 江安校区是本次交付的唯一运营场景，但系统概念、数据和接口须保留 `Scene`（文旅场景）维度，禁止将“校园”作为系统固定前提。
- 技术实现以模块化单体为目标；游客小程序、Web 管理端、硬件和 AI 服务均通过后端受控访问业务数据。

## 2. 产品目标与本次交付边界

### 2.1 一句话目标

让游客佩戴已绑定的 RFID/NFC 手环，在江安校区完成真实点位打卡，并在微信小程序中看到地图、进度、记录与照片；AI 只能依据这些真实记录回答问题、推荐下一站并生成个性化游记。

### 2.2 必须交付的闭环

```text
微信登录与绑卡
    → 游客携带手环到指定点位
    → RFID/NFC 设备上传打卡事件
    → 后端认证、校验、去重并保存
    → 小程序更新进度、地图与记录
    → AI 读取受控的真实路线
    → 输出路线问答 / 点位推荐 / 个性化游记
```

### 2.3 本次必须实现

| 模块 | MVP 要求 | 可被验收的结果 |
|---|---|---|
| 场景与点位 | 预置江安校区场景、核心点位、图片与基础介绍 | 地图和详情均能展示对应内容 |
| 身份与卡片 | 微信登录；一账号可有多张有效卡；一张卡同一时刻只属于一位用户 | 绑定、主要卡、查看状态和解绑均符合规则 |
| 真实打卡 | 设备读取 RFID/NFC 卡；后端认证、去重、保存并返回结果 | 一次有效刷卡只形成一条有效打卡记录 |
| 小程序导览 | 首页、地图、点位详情、打卡记录、我的 | 已打卡/未打卡与完成度一致 |
| 照片 | 仅明确标识的拍照点；照片与有效打卡关联 | 只有有权用户可在记录中查看自己的照片 |
| 推荐 | 根据已打卡、未打卡、剩余时间和偏好进行规则推荐 | 同一输入的推荐结果可解释、可复现 |
| AI 服务 | 查询真实路线/进度、点位问答、下一站推荐、生成游记 | 输出不含未打卡路线事实或虚构点位经历 |
| 运营/管理 | 独立 Web 管理端，同时提供比赛展示看板和完整运营管理 | 可展示真实统计，并管理点位、设备、卡片、绑定、打卡、媒体和 AI 内容 |
| 演示 | 一条从登录到生成游记的现场演示链路 | 可在真实设备和小程序中连续完成 |

### 2.4 明确不作为本次必需交付

- UHF RFID 隐式打卡、停留判断与轨迹拟合；
- 自动视频生成、用户画像、社交匹配、RFID 姿态识别；
- 多景区运营平台、复杂实时导航、多 Agent 协作；
- 摄像头人脸识别、无感追踪或任何未告知的影像采集。

这些方向可在答辩中作为后续演进，不得挤占 MVP 闭环的联调与稳定性时间。

## 3. 角色、对象与通用定义

| 名称 | 定义 | 本次使用方式 |
|---|---|---|
| Scene | 可运营的文旅场景 | 本次为“四川大学江安校区”一条记录 |
| User | 通过微信登录的游客账号 | 浏览个人数据、绑定卡片、使用 AI 服务 |
| Card | RFID 卡片或手环的物理标识 | 仅作为现场识别媒介，不承担业务逻辑 |
| CardBinding | 用户与卡片在一段时间内的绑定关系 | 用户可有多张有效卡；卡片同一时刻仅有一个有效用户 |
| Device | 安装在点位的硬件终端 | 上传感知事件、接收成功/失败反馈 |
| Spot | 可被导览、介绍或打卡的文旅点位 | 隶属一个 Scene；可标记为普通、景观或拍照点 |
| VisitSession | 用户的一次游览 | 聚合该次游览的有效打卡、进度与游记 |
| CheckinEvent | 一次刷卡或感知事件 | 保存校验结果；只有有效事件纳入游览路线 |
| CheckinMedia | 与有效打卡关联的照片 | 仅在用户知情和授权的拍照点产生 |
| AIGeneration | AI 生成的游记、分享文案等结果 | 保存生成依据、版本和内容，便于回顾 |

## 4. 用户主流程

### 4.1 进入与绑卡

1. 游客进入微信小程序并完成微信授权登录。
2. 系统展示卡片列表和主要卡片状态：未绑定、已绑定、已失效或需更换。
3. 未绑定用户优先通过手机 NFC 感应读取卡片 UID；感应不可用时，用户输入卡片 UID 作为降级方式，并提交绑定请求。
4. 后端确认用户、卡片和绑定状态均符合规则后，创建有效绑定。
5. 首页显示主要卡片的脱敏标识、有效卡片数量和本次游览入口。

绑定本身不等于打卡，也不应自动创建游览记录。

### 4.2 现场打卡

1. 游客到达已部署设备的点位，在清晰的提示下刷手环。
2. 设备读取卡 UID，生成唯一 `event_id`，携带设备、点位和事件信息上传后端。
3. 后端校验设备、点位、卡片绑定、场景归属和重复事件。
4. 有效时保存打卡，更新对应游览会话；设备提供成功反馈。
5. 无效时不写入有效路线，设备提供失败或引导反馈；小程序可在刷新或实时通知后看到最新状态。

### 4.3 游览查看与 AI 服务

1. 游客在首页查看本次进度与最近一次打卡。
2. 在地图上区分已打卡、未打卡、拍照点和点位简介。
3. 在记录页按时间查看已打卡点位、顺序、方式、照片和完成度。
4. 游客可请求规则推荐，或通过 AI 用自然语言查询路线、点位和下一站。
5. 游客要求生成游记时，系统只读取本次会话中的有效路线与授权素材，经固定工作流生成后保存为个人内容。

## 5. 核心业务规则

### 5.1 账号与卡片绑定

| 规则编号 | 规则 |
|---|---|
| BIND-01 | 一个账号在任一时刻可以存在多条有效卡片绑定，并可指定一张主要展示卡。 |
| BIND-02 | 一张卡在任一时刻最多绑定一个账号。 |
| BIND-03 | 绑定卡已被其他有效账号占用时，系统必须拒绝绑定，不能静默覆盖原关系。 |
| BIND-04 | 解绑或更换卡片必须使原绑定立即失效，并保留历史绑定审计；历史打卡不得因解绑而删除。 |
| BIND-05 | 设备上传的卡 UID 不代表可展示的用户身份；后端负责映射绑定关系，前端不得展示完整 UID。 |
| BIND-06 | 未绑定、已失效或归属不明的卡片刷卡，均不得生成有效打卡。 |
| BIND-07 | NFC 读取必须先进行设备与能力检测；任何读卡失败、权限拒绝或机型不支持时，都必须无阻断地切换至手动输入流程。手机 NFC 是便利能力，不能成为完成绑卡的唯一前提。 |
| BIND-08 | 卡 UID 是识别符而非登录凭据。手动输入 UID 时，除格式和占用校验外，必须具有一项防误绑/冒领校验；不得仅凭知道一个 UID 就允许把该卡绑定到任意账号。 |

**已确认的绑定方式：**小程序优先使用手机 NFC 读取卡片 UID；手机不支持、权限受限或感应失败时，允许用户输入 UID 进行绑定。两种方式均须由后端校验 UID 是否存在、是否可用以及是否已被占用。工作人员可仅通过管理后台协助异常处理，不作为游客的主流程。

**已确认的手输 UID 校验：**每张实体卡随卡发放唯一的一次性激活码。手输 UID 时，用户还必须输入该激活码；后端以原子方式校验卡片状态、UID、激活码和绑定占用状态，绑定成功后立即将激活码标记为已使用。错误、已使用或与卡片不匹配的激活码不得创建绑定。激活码的生成、存储、重置和脱敏展示仅允许在后端和管理后台处理。

### 5.2 设备与点位

| 规则编号 | 规则 |
|---|---|
| DEVICE-01 | 每台设备拥有唯一 `device_id`，并在后端登记所属 Scene 和允许服务的 Spot。 |
| DEVICE-02 | 设备只能为已授权的点位上传打卡；`device_id` 与 `spot_id` 不匹配时，事件必须拒绝。 |
| DEVICE-03 | 硬件只负责读卡、组装事件、上传和本地反馈；不得在设备端判断用户、路线、推荐或 AI 结果。 |
| DEVICE-04 | 所有设备请求须经设备级认证；认证凭据不得写入小程序或公开前端代码。 |
| DEVICE-05 | 设备短暂断网可重传未确认事件；重传不得形成重复有效打卡。 |
| DEVICE-06 | 一台设备可为任意数量的合法卡片读取和上报事件；设备不保存“卡片属于哪位用户”的绑定关系，绑定关系仅由后端维护。 |

### 5.3 打卡事件、有效性与去重

设备上报事件的最低业务字段为：`device_id`、`spot_id`、`card_uid`、`checkin_type`、`event_id`，可附带 `device_time`。字段名称和签名方式将在接口文档中定稿。

| 规则编号 | 规则 |
|---|---|
| CHECKIN-01 | 后端收到事件后，依次校验设备认证、设备/点位关系、场景归属、卡片有效绑定、`event_id` 唯一性和必要字段。 |
| CHECKIN-02 | `event_id` 是设备端生成的幂等键；同一设备对同一 `event_id` 的重复上报必须返回同一处理结果，不得重复写入。 |
| CHECKIN-03 | 只有通过校验的事件才标记为有效，并进入进度、路线、推荐和 AI 数据来源。无效事件可保留最小审计信息，但不可对游客形成路线事实。 |
| CHECKIN-04 | 后端服务时间是排序与业务判定的可信时间；`device_time` 用于诊断、展示或离线补传参考，不能单独作为可信依据。 |
| CHECKIN-05 | 有效事件处理结果至少包括：成功、重复成功、设备未授权、点位不匹配、卡未绑定/失效、参数非法、服务暂不可用。设备据此提供明确且不泄露个人信息的反馈。 |
| CHECKIN-06 | 用户不能通过小程序自行新增、修改或删除打卡事实；手动数据纠正仅允许在管理权限和审计流程下进行。 |

**已确认规则：**同一用户在同一 Spot 的第二次及以后有效刷卡，可保留为事件审计记录；游览完成度按点位去重，路线和默认记录展示首次到达。这样既保留真实操作记录，又避免完成度重复累计。

**实施参数 CHECKIN-B：**同卡同点位的短时提示/限流窗口，用于降低误触和连续刷卡噪声。窗口长度由硬件实测确定；无论最终阈值如何，完成度均按 Spot 去重，且 `event_id` 幂等规则不变。

### 5.4 游览会话与进度

`VisitSession` 是“本次游览”的数据边界，决定首页进度、记录筛选、AI 路线和游记使用的打卡集合。

| 规则编号 | 规则 |
|---|---|
| SESSION-01 | 每个游览会话只属于一个用户和一个 Scene。 |
| SESSION-02 | 有效打卡只能归入该用户在该 Scene 的一个游览会话。 |
| SESSION-03 | 进度按“会话内已访问的不同可打卡 Spot 数 / 场景参与统计的 Spot 总数”计算；重复刷卡不重复计数。 |
| SESSION-04 | 已打卡/未打卡状态应由有效打卡事实实时或准实时计算，不能仅由前端本地缓存决定。 |
| SESSION-05 | AI 查询“今天”“本次”路线时，必须明确使用的会话时间范围，并将无有效打卡的情况如实返回。 |

**已确认会话生命周期：**以中国标准时间（`Asia/Shanghai`）的自然日为一个打卡周期。用户在同一 Scene、同一自然日的全部有效打卡归入同一 `VisitSession`；次日首条有效打卡自动创建新会话。第一版不要求用户手动开始或结束会话，也不区分同日的多次游览。

### 5.5 地图、点位与记录展示

| 规则编号 | 规则 |
|---|---|
| MAP-01 | 地图采用“江安校区卡通静态底图 + 点位比例坐标”的 MVP 方案。 |
| MAP-02 | Spot 至少具备名称、所属 Scene、坐标、类型、简介、图片、是否可打卡、是否拍照点等可展示信息。 |
| MAP-03 | 地图点位至少区分已打卡、未打卡与拍照点；视觉状态应由后端返回的有效数据驱动。 |
| MAP-04 | 点位详情可展示公开介绍、素材、当前打卡状态和推荐信息；不展示其他游客的个人打卡或照片。 |
| RECORD-01 | 打卡记录默认展示当前会话的有效点位、首次到达时间、到达顺序、打卡方式与有权访问的照片。 |
| RECORD-02 | 历史游览与当前游览必须可区分；不同会话的数据不能混入同一条“本次路线”。 |

### 5.6 拍照与隐私

| 规则编号 | 规则 |
|---|---|
| MEDIA-01 | 摄像头仅部署在清晰标识的拍照点；游客在拍照前应知悉采集目的和使用范围。 |
| MEDIA-02 | 照片只可与一个通过校验的打卡事件关联，不能被当作独立的无感轨迹数据。 |
| MEDIA-03 | 照片访问需经过后端权限检查，仅照片所属用户及被授予管理权限的人员可访问。 |
| MEDIA-04 | 不进行人脸识别、身份推断或无感追踪；不得将照片用于超出打卡留念的目的。 |
| MEDIA-05 | 数据库保存媒体 URL 与必要元数据；图片文件存放于受控对象存储，并使用受权限保护的访问方式。 |

**已确认的拍照边界：**采用“明确标识拍照点 + 有效打卡事件关联 + 事前告知”的方案。拍照触发方式、保存时长和删除入口仍需在隐私说明与接口设计中明确；初赛前不得以“默认同意”代替明确告知。

### 5.7 规则推荐

第一版的推荐由确定性规则服务产生，AI 负责理解自然语言、调用服务并解释结果，而不是绕开规则自行编造推荐。

输入至少包括：当前会话已打卡点位、未打卡点位、用户剩余时间（如提供）、偏好标签（如提供）和各点位元数据。

| 规则编号 | 规则 |
|---|---|
| RECO-01 | 默认优先从当前 Scene 的未打卡、可访问点位中选择。 |
| RECO-02 | 无法确定实时位置时，系统只能将“最近一次有效打卡点位”作为位置近似，并明确不声称已获取实时定位。 |
| RECO-03 | 推荐规则应支持点位标签、推荐停留时间、最近打卡点位关系和剩余时间等可解释因素。 |
| RECO-04 | 剩余时间不足以覆盖推荐停留时间时，应降低该点位优先级或明确提示时间风险。 |
| RECO-05 | 推荐结果应提供简短理由，例如“未打卡、适合拍照、建议停留 15 分钟”，不能编造距离、开放活动或实时拥挤度。 |

**已确认的位置边界：**第一版不申请微信实时位置权限；推荐仅使用最近一次有效打卡点位作为位置近似，并在产品表述中避免声称已经获取游客实时位置。

### 5.8 AI Agent 与内容生成

AI 是受控上层服务，不是事实来源，也不能替代后端权限判断。

第一版 Agent 可调用的受控工具：

- `get_current_route`：取得指定当前/历史游览会话的真实路线；
- `get_checkin_progress`：取得会话完成度和已/未打卡摘要；
- `get_spot_detail`：取得公开点位资料；
- `list_unvisited_spots`：取得当前会话的未打卡点位；
- `recommend_next_spots`：调用规则推荐服务；
- `generate_story_package`：触发固定游记生成工作流；
- `get_generation_history`：读取当前用户有权访问的历史生成内容。

| 规则编号 | 规则 |
|---|---|
| AI-01 | 小程序不得直接保存或调用 Dify、模型服务等密钥；完整链路必须经过后端身份验证。 |
| AI-02 | Agent 只能调用受控工具，不得直接查询数据库、读取其他用户数据或修改业务事实。 |
| AI-03 | 当用户询问路线或进度时，回答必须以工具返回的有效数据为准；缺少数据时应明确说明，不能补造经历。 |
| AI-04 | AI 不得增加未打卡点位、编造天气/活动/人物/价格，或未经确认向外部平台发布内容。 |
| AI-05 | 游记生成使用固定工作流：读取真实路线与点位资料 → 规划叙事 → 生成游记/分享文案 → 事实检查 → 必要时修复 → 保存。 |
| AI-06 | 事实检查至少验证：输出中的游览点位均来自本次会话有效打卡；点位介绍来自受控知识资料；没有不支持的时事或现实断言。 |
| AI-07 | 每次生成应保存所属用户、所属会话、生成类型、生成时间、依据的打卡集合/版本和最终内容，以支持历史查看与问题追溯。 |

## 6. 小程序页面职责与最低验收点

| 页面 | 核心职责 | 最低验收点 |
|---|---|---|
| 登录与绑卡 | 登录、显示/变更绑定状态 | 未绑定用户不能被误显示为可打卡；绑定冲突有明确提示 |
| 首页 | 建立本次游览概览与任务入口 | 显示本次完成度、当前卡状态、最近打卡和导览/记录/推荐入口 |
| 校园导览 | 在卡通地图中探索点位 | 点位坐标正确；点击可见详情与个人打卡状态 |
| 点位详情 | 提供点位知识与到访状态 | 公共资料、状态、图片和拍照点标识一致 |
| 打卡记录 | 回顾路线和照片 | 顺序、时间、完成度与后端有效事件一致 |
| 推荐点位 | 展示规则服务推荐 | 有明确理由、停留建议、打卡状态；没有“伪实时”信息 |
| AI 智能体 | 自然语言访问受控能力 | 路线、进度、推荐、游记的依据可追溯到真实数据 |
| 我的 | 管理个人数据与回顾 | 卡片、历史游览、游记、照片和隐私入口清晰 |

## 7. 异常与边界处理

| 场景 | 面向游客/设备的处理 | 数据处理原则 |
|---|---|---|
| 未绑卡刷卡 | 设备提示需先绑定或咨询工作人员，不显示其他用户信息 | 记录最小审计信息；不产生有效路线 |
| 卡片已失效/更换 | 设备提示卡片不可用 | 不产生有效打卡 |
| 重传同一事件 | 设备收到与首次一致的成功/失败处理结果 | 以 `event_id` 幂等，不重复写入 |
| 同点重复刷卡 | 设备可提示已记录 | 事件可留作审计；完成度不重复累加，路线默认展示首次到达 |
| 设备离线 | 提示设备状态或稍后重试；允许安全重传 | 后端仍以幂等校验为准 |
| AI 无有效路线 | 明确说明尚未获得本次真实打卡 | 不生成包含虚构到访事实的游记 |
| 用户无照片权限 | 页面不显示资源或返回无权限提示 | 后端拒绝资源访问，不能只靠前端隐藏 |
| 点位资料缺失 | 展示受控兜底文案，避免编造 | 在后台/监控中标记待补全 |

### 7.1 Web 管理端职责

Web 管理端是本次正式交付的一部分，面向项目成员和运营管理员，不面向普通游客。它同时承担两类职责：

- 比赛展示：场景总览、当日游客与打卡数据、点位热度、设备状态、最新打卡、路线和照片等可视化；
- 运营管理：点位与素材、地图坐标、设备、卡片与激活码、绑定记录、游览会话、打卡事件、照片、AI 生成内容和管理员审计。

展示和管理共用同一套真实后端数据，不设置另一套数据库，也不得在前端伪造“实时数据”。比赛所需的测试/演示数据必须明确标识环境或数据来源。Django Admin 仅作为系统管理员的内部维护和故障兜底入口，不作为比赛主展示页面。

管理端应实施角色权限、敏感字段脱敏、高风险操作二次确认和审计。普通运营人员不得查看完整卡 UID、激活码明文、设备密钥或其他不属于其职责的数据。

## 8. 最终现场演示验收链路

演示前必须以一位测试游客完整走通以下步骤，并记录结果：

1. 使用未绑定账号登录，成功绑定一张测试手环；
2. 首页正确显示卡片状态，初始进度为零；
3. 在至少三个江安校区点位真实刷卡，其中包含一个拍照点；
4. 每次刷卡设备都有明确反馈，后端只保存一条有效记录；
5. 小程序地图、首页和记录页都能反映相同的已打卡集合、顺序与完成度；
6. 游客可查看有权访问的打卡照片；
7. 规则推荐返回未打卡点位，并说明推荐原因；
8. AI 正确回答“我今天/本次去了哪里”，结果与打卡记录一致；
9. AI 生成游记与分享文案，内容只引用实际打卡点位和受控点位资料；
10. 在 Web 管理端展示真实统计，并可查询该设备、卡片绑定（脱敏）和打卡结果；
11. 对重复事件、未绑定卡和 AI 无路线三类异常至少各验证一次。
