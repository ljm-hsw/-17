# 游迹织梦：后台管理系统前端 v1 设计

> 文档状态：已确认，进入实施计划前的团队评审版  
> 确认日期：2026-07-22  
> 目标分支：`test`  
> 技术栈：Vue 3 + TypeScript + Vite + Pinia + Vue Router + Element Plus + ECharts  
> 关联文档：[管理端长期产品愿景](2026-07-21-admin-web-design.md)、[基础数据与 API 设计](2026-07-21-base-data-api-design.md)

## 1. 目标

首版后台管理系统同时服务两个场景：

1. 比赛展示人员能够用真实后端数据说明“卡片—设备—点位—打卡—游览—运营观察”的完整闭环；
2. 团队成员能够在统一、可维护的界面中管理点位、路线、设备、卡片和绑定，并查询游览、打卡与审计记录。

首版以“可真实联调、可比赛展示、方便团队分工”为成功标准，不追求一次实现长期产品愿景中的全部模块。Web 前端不保存业务数据库，不直接读取 PostgreSQL；所有业务数据均来自 Django 管理 API。

## 2. 已确认决策

- 采用“模块化 Demo 主线”：先建立登录、整体布局、API Client、权限和通用交互，再实现全部已受后端支持的页面。
- 采用“文化科技融合”视觉方向：暖米白、江安绿和少量暖橙，展示看板突出品牌感，运营页面优先保证可读性。
- 使用按业务分组的固定侧边栏、场景与管理员顶部栏以及统一内容区。
- 运营页面统一使用“筛选列表 + 详情抽屉 + 独立复杂表单 + 高风险确认”的交互范式。
- 允许为真实联通补充少量 Django 管理端登录和查询能力，但不扩大业务范围。
- 首版不实现照片管理、AI 内容管理和管理员角色管理；导航仅以不可点击的“规划中”入口说明方向。
- PostgreSQL、Docker 等较重集成验证暂缓，前端单元测试、类型检查、构建和轻量联调必须执行。

## 3. 用户与权限

### 3.1 首版角色

首版前端按 Django 返回的权限编码控制路由、按钮和字段；后端继续执行最终授权。

| 角色 | 首版主要能力 |
|---|---|
| 展示人员 | 只读展示看板、场景、点位、设备状态和打卡数据 |
| 内容运营 | 管理场景、点位、媒体和路线 |
| 设备运营 | 管理设备归属、状态和凭据 |
| 数据管理员 | 管理卡片、激活码、绑定以及业务数据查询 |
| 超级管理员 | 拥有首版全部能力，供团队 Demo 联调 |

前端隐藏或禁用无权限入口是体验优化，不是安全边界。任何写操作都必须由 Django 再次检查模型权限。

### 3.2 团队测试账号

- 团队共享本地测试账号使用用户名 `demo_admin`，初始本地演示密码为 `TravelWeave-Demo-2026!`。
- `backend/.env.example` 使用 `DEMO_ADMIN_USERNAME`、`DEMO_ADMIN_PASSWORD` 和 `DEMO_ADMIN_NICKNAME` 保存公开的本地演示默认值；后台 README 记录初始化命令和“禁止生产使用”说明。
- `.env.local`、真实密码和生产凭据不得提交到 Git。
- 演示数据命令负责幂等创建或更新测试管理员，不在登录视图中硬编码密码。
- 测试管理员标记为 `is_demo=True`；当 `DEBUG=False` 时，管理端登录接口拒绝测试账号。

## 4. 首版范围

### 4.1 页面与路由

| 路由 | 页面职责 |
|---|---|
| `/login` | 管理员登录、环境标识、测试账号说明 |
| `/dashboard` | 指标、趋势、排行、设备状态、最新事件和演示模式 |
| `/scenes` | 场景资料、地图底图和点位空间概览 |
| `/spots` | 点位列表、创建、编辑、发布、停用和点位媒体 |
| `/routes` | 路线列表、基础资料和有序路线点位 |
| `/devices` | 设备登记、归属、状态、详情、事件和密钥轮换 |
| `/cards` | 单卡登记、批量导入预览、激活码和绑定查询 |
| `/users` | 脱敏用户查询、有效卡片、绑定历史和强制解绑 |
| `/visits` | 自然日游览列表、详情和关联打卡 |
| `/checkins` | 打卡事件筛选、状态、失败原因和关联对象 |
| `/audit-logs` | 只读操作审计查询 |
| `/403` | 已登录但无访问权限 |
| `/404` | 不存在的前端路由 |

照片与 AI 仅在侧边栏底部显示一个不可点击的“规划中”入口，不创建空白页面，不伪造接口数据。

### 4.2 明确不在首版范围

- 打卡照片审核与删除；
- AI 生成任务、内容审核和重试；
- 管理员账号、角色和权限编辑；
- WebSocket 实时推送；
- 拖拽式大屏、任意报表与多租户；
- 前端直接控制设备或直接访问数据库；
- 为填满页面而生成无法解释的随机业务数据。

    │   └── 展示看板
    ├── 内容运营
    │   ├── 场景与地图
    │   ├── 点位管理
    │   └── 游览路线
    ├── 物联运营
    │   ├── 设备管理
    │   └── 卡片与激活码
    ├── 数据查询
    │   ├── 用户与绑定
    │   ├── 游览记录
    │   └── 打卡事件
    └── 系统
        ├── 操作审计

侧边栏保持稳定，按权限过滤入口。顶部栏只承载当前 Scene、测试环境标识、演示模式、帮助入口和管理员菜单，不堆放业务筛选器。

## 6. 视觉系统

### 6.1 设计语言

- 主背景：暖米白与极浅灰绿，避免纯白页面缺少项目辨识度；
- 主色：江安绿，用于导航选中、主按钮、成功状态和图表主系列；
- 强调色：暖橙，用于核心指标、高风险提示和少量品牌细节；
- 文字：深墨绿或深灰，正文对比度至少 4.5:1；
- 卡片：白色或高不透明度浅色，细边框和轻阴影；毛玻璃只用于登录页、看板或弹层，不用于大面积数据表格；
- 图标：统一使用同一套 SVG 图标，不使用 emoji 作为功能图标。

### 6.2 布局与响应式

- 1440px：完整侧边栏、双列看板和宽数据表格；
- 1024px：保留侧边栏，看板降为单列或紧凑双列；
- 768px：侧边栏折叠为抽屉，筛选器换行，详情抽屉占更大比例；
- 375px：保证登录、查看与关键确认可用；复杂表格采用固定关键列与横向滚动，不强求完整运营体验。

交互动画使用 `transform` 和 `opacity`，时长 150–300ms，并尊重 `prefers-reduced-motion`。所有按钮、链接和图标按钮具有可见 hover、active 和 focus 状态。

## 7. 页面与组件设计

### 7.1 展示看板

首屏固定展示四项现有后端可准确提供的指标：

- 今日游客；
- 当前范围内的有效打卡；
- 在线设备与设备总数；
- 当前有效绑定卡片。

看板其余模块包括近七日打卡趋势、点位热度、江安校区空间概览、最新打卡事件和设备运行状态。每个模块独立加载和报错；局部失败不得让其他模块白屏，也不得将失败显示成数字 `0`。

第一版采用 10 秒摘要/设备轮询和 5 秒事件流轮询，支持手动刷新。页面必须显示最后更新时间。进入演示模式后隐藏写操作、弱化侧边栏并允许浏览器全屏，但不改变账号真实权限。

### 7.2 运营页面通用模式

1. 页面标题区说明对象、当前场景和主要创建操作；
2. 汇总卡片只在确有清晰口径时使用；
3. 筛选区负责关键词、Scene、状态、日期和对象关联筛选；
4. 表格负责比较、排序、分页和进入详情；
5. 详情抽屉用于快速查看关联数据和轻量编辑；
6. 创建点位、路线编排、批量导卡等复杂流程使用独立页面或分步弹窗；
7. 停用、强制解绑和密钥轮换使用统一高风险确认弹窗，要求明确动作、影响、原因和二次确认。

### 7.3 一次性敏感数据

设备首次创建或轮换密钥、卡片生成激活码后：

- 明文仅存在于当前 API 响应和当前展示弹窗；
- 弹窗明确提示“关闭后无法再次查看”；
- 提供复制按钮和复制成功反馈；
- 不写入浏览器持久化状态、前端日志、埋点或错误报告；
- 关闭前要求用户确认已保存，不提供“再次获取明文”。

## 8. 前端工程架构

```text
admin-web/src/
├── components/       # 布局、状态、表格、确认框等跨业务组件
├── features/         # dashboard、spots、devices、cards 等领域模块
├── views/            # 路由页面容器
├── services/         # API Client 与领域 API
├── stores/           # 登录、权限、当前 Scene 和必要页面状态
├── router/           # 路由表、权限元数据和守卫
├── types/            # API 契约与领域类型
├── utils/            # 时间、状态、错误映射等纯函数
└── styles/           # Token、Element Plus 覆盖和全局响应式规则
```

约束如下：

- 页面不直接拼接 API URL；
- API 服务不操作 DOM，也不弹出 UI 消息；
- Store 只保存跨页面共享状态，不代替服务层；
- 业务组件通过明确 Props、Emits 和类型工作；
- 通用组件不引用具体业务接口；
- 单文件组件过大时按列表、筛选、抽屉和表单拆分，不在一个文件中承载整个模块。

## 9. 登录、权限与会话

### 9.1 管理端认证接口

为前端补充以下管理接口：

```text
POST /api/v1/management/auth/login
GET  /api/v1/management/auth/me
POST /api/v1/auth/refresh
```

登录使用 Django 用户名和密码校验，只允许 `is_active=True` 且 `is_staff=True` 的账号。登录和 `me` 返回：

```json
{
  "id": "user-uuid",
  "username": "demo_admin",
  "nickname": "演示管理员",
  "is_superuser": true,
  "is_demo": true,
  "permissions": ["scenes.change_spot", "iot.change_device"]
}
```

实际响应继续包裹在项目统一的 `{ "data": ..., "request_id": ... }` 结构中。登录响应同时返回 access 与 refresh token。

### 9.2 前端会话策略

- access 与 refresh token 首版保存在 `sessionStorage`，关闭浏览器后清除；
- API Client 自动添加 Bearer access token；
- access 过期后只允许一次刷新请求，多个并发失败请求复用同一个刷新过程；
- 刷新成功后重放安全的读取请求；写请求仅在确认后重放，避免重复创建或高风险操作；
- 刷新失败时清除会话并回到登录页，保留原目标路由用于重新登录后返回；
- 退出登录清除 token、管理员资料、权限和 Scene 状态。

## 10. API Client 与数据流

API Client 统一处理：

- `VITE_API_BASE_URL`；
- JWT 请求头；
- `{ data, meta, request_id }` 响应解包；
- 分页元数据；
- 超时与网络错误；
- 后端字段错误和业务错误；
- `request_id` 保存到标准错误对象供界面复制。

典型读取数据流：

```text
路由页面 → 领域 Service → API Client → Django 管理 API
        ← 页面状态  ← 类型化结果  ← data/meta/request_id
```

典型高风险写入数据流：

```text
用户点击 → 权限检查 → 展示影响与原因表单 → 二次确认
         → Service 提交 confirm + reason → 后端权限/业务/审计
         → 成功后局部刷新；失败时保留输入并展示 request_id
```

## 11. 必要的后端联通补充

以下调整属于现有管理能力的前端可用性补充，不新增独立业务模块：

1. 新增管理端登录和当前管理员权限接口；
2. Scene 管理响应补充 `map_image_url`；
3. Spot 管理详情补充简介、正文、知识内容、坐标、标签、停留时长、拍照点和媒体；
4. Spot 媒体提供当前点位的读取能力；
5. 用户、卡片、设备、游览和打卡列表补充页面需要的服务端筛选；
6. 看板趋势、排行和设备状态统一接受 `scene_id` 与日期范围，避免混合不同口径；
7. 游览详情返回或允许查询该次游览关联的打卡事件；
8. 审计响应补充可展示的脱敏管理员名称，仍保留 actor ID；
9. 所有列表保持 `data.items + meta.page/page_size/total` 结构；
10. 敏感值继续只返回掩码、指纹或一次性明文。

前端不得通过游客接口拼装管理页面，也不得为了绕过管理 API 缺口直接调用 Django Admin 或数据库。

## 12. 页面状态与错误处理

所有看板模块和列表页面必须区分：

- 首次加载；
- 刷新中；
- 正常数据；
- 空数据；
- 筛选无结果；
- 无权限；
- 网络或服务失败；
- 局部数据不可用。

错误处理规则：

- 字段校验错误显示在对应表单项下并保留输入；
- UID 重复、场景与点位不匹配等业务冲突显示明确中文信息；
- 401 进入刷新流程，403 进入无权限状态，404 进入对象不存在状态；
- 网络失败不显示为零数据；
- 创建、停用、解绑、导入确认和密钥轮换失败后不自动重复提交；
- 看板局部接口失败时保留其他成功模块；
- 所有异常提示允许复制 `request_id`，便于前后端联合排查。

## 13. 测试与验证

### 13.1 自动化测试

Vitest 覆盖：

- API 响应解包、错误映射和 token 刷新队列；
- Auth Store、Scene Store 和退出清理；
- 路由登录守卫与权限守卫；
- 加载、空数据、无结果、失败和无权限状态组件；
- 高风险确认表单；
- 一次性激活码或设备密钥弹窗；
- 至少一个代表性列表模块和一个表单模块。

测试通过服务层 mock 构造确定数据，不在组件中启用随机 Demo 数据。

### 13.2 每次交付检查

```bash
npm test
npm run type-check
npm run build
```

本机使用 Node.js 24 LTS。当前机器系统默认 Node 25 存在动态库问题，运行 npm 命令前使用 `/opt/homebrew/opt/node@24/bin`。

### 13.3 联调冒烟

使用 `demo_admin` 在轻量开发环境完成：

1. 登录并进入看板；
2. 查看真实演示数据与最新事件；
3. 编辑并保存一个点位；
4. 登记或批量导入卡片并生成一次激活码；
5. 查看用户绑定并完成带原因的强制解绑；
6. 登记设备并完成带原因的密钥轮换；
7. 查询游览、打卡和对应审计记录；
8. 刷新页面验证会话与权限恢复；
9. 模拟 401、403、网络失败和局部看板失败。

PostgreSQL、Docker 和硬件端到端验证不作为本阶段前端提交的阻塞项，后续正式联调再执行。

## 14. 首版验收标准

首版完成必须同时满足：

1. 管理员使用真实用户名与密码登录，JWT 刷新、退出和权限守卫可用；
2. 团队测试账号由环境变量与演示数据命令规范创建，生产环境拒绝 Demo 账号；
3. 展示看板使用真实 Django 数据并明确时间、场景和最后更新时间；
4. 全部首版路由可访问并覆盖必要页面状态；
5. 点位、路线、设备和卡片的现有写接口具有可用表单；
6. 用户、绑定、游览、打卡和审计可完成跨对象追踪；
7. 高风险操作需要权限、原因、二次确认并产生审计；
8. 完整 UID、激活码历史明文、设备密钥和 token 不出现在普通页面或日志；
9. 1440px 比赛屏和常见笔记本布局清晰，窄屏无不可恢复的横向溢出；
10. 基础无障碍、测试、类型检查和生产构建通过；
11. 前端没有用写死数据掩盖接口失败或未实现能力；
12. 新队友可以依据 README、本文档和模块边界独立承担一个业务模块。
